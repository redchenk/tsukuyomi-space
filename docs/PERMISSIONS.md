# 权限模型

月读空间目前有两条身份链路：普通用户链路使用 `users` 表和 `/api/auth/*`，管理员终端链路使用 `admins` 表和 `/api/admin/login`。两类 token 分开保存，前端分别使用 `tsukuyomi_token` 和 `admin_token`。

## 角色边界

| 角色 | 来源 | 能力 | 禁止 |
|------|------|------|------|
| `user` | `users.role` | 登录个人账号；创建普通文章；发布、回复、点赞留言；访问个人中心和需要登录的互动功能。 | 访问 `/api/admin/*`；发布公告分类文章；管理其他用户、文章、留言、站点设置。 |
| `admin` | `users.role` 或 `admins.role` | 作为内容管理员，可访问管理员终端 API；管理文章、留言、友链、用户列表、站点设置和统计。`users.role = admin` 也可在前台执行管理员级内容操作。 | 不应绕过审计直接操作生产数据库；不能假定拥有根级系统权限。 |
| `super_admin` | `admins.role` | 初始管理员账号默认角色。当前代码把它作为管理员终端最高角色，具备 `admin` 的全部能力，并预留给后续更敏感操作，如管理员账号管理、权限变更、危险维护动作。 | 不用于普通用户登录链路；不得把 `super_admin` token 暴露给浏览器以外的第三方服务。 |

## 服务端校验

- `authenticateToken` 只校验普通 JWT 是否有效，并把载荷写入 `req.user`。
- `requireAdmin` 只允许 `role === 'admin'` 或 `role === 'super_admin'` 继续访问。
- `/api/admin/login` 使用独立的 `admins` 表签发管理员 token，载荷包含 `scope: 'admin'`。
- 普通用户 token 即使登录成功，也不能访问 `/api/admin/*`。
- 前端显示隐藏只用于体验，不能作为安全边界；所有权限必须以后端校验为准。

## 数据操作边界

- 普通用户只能修改自己创建的内容，公告类内容除外。
- 管理员终端可以查看和管理全站文章、留言、用户、友链、站点设置和访问统计。
- 删除用户时，当前逻辑禁止删除管理员用户或默认管理员账号。
- 数据库迁移由应用启动时自动执行；生产部署前必须先完成 SQLite 备份。

## 后续扩展建议

- 如果引入管理员账号管理，只有 `super_admin` 可以创建、禁用或降级管理员。
- 如果引入审计日志，管理员和 `super_admin` 的写操作都应记录操作者、目标资源、时间和请求来源。
- 如果拆分更细权限，保持“默认拒绝”，只给明确需要的角色开放具体 API。
